## 0. Сначала — что такое «бэкенд» и HTTP (на пальцах)

Бэкенд — это программа, которая **крутится на сервере и отвечает на запросы**.
Она ничего не рисует на экране. Она получает запрос, что-то делает (лезет в базу,
считает, проверяет права) и возвращает **ответ** — обычно в формате JSON.

Общение идёт по протоколу **HTTP**. Запрос состоит из:

| Часть | Пример | Что значит |
|---|---|---|
| **Метод (глагол)** | `GET`, `POST`, `PUT`, `DELETE` | что хотим сделать: получить / создать / обновить / удалить |
| **URL (путь)** | `/api/ping`, `/api/grades/123` | к какому ресурсу обращаемся |
| **Заголовки** | `Authorization: Bearer ...` | метаданные: кто я, какой формат принимаю |
| **Тело (body)** | `{"value": 5}` | данные (есть у POST/PUT, обычно нет у GET) |

Ответ состоит из:

| Часть | Пример | Что значит |
|---|---|---|
| **Код статуса** | `200`, `201`, `400`, `401`, `404`, `500` | чем всё кончилось (см. таблицу ниже) |
| **Заголовки** | `Content-Type: application/json` | метаданные ответа |
| **Тело** | `{"message":"pong"}` | сами данные |

Коды статусов, которые надо знать на собеседовании **наизусть**:

- **2xx — успех.** `200 OK` (всё ок), `201 Created` (создали новый ресурс), `204 No Content` (ок, но возвращать нечего).
- **4xx — виноват клиент.** `400 Bad Request` (кривые данные), `401 Unauthorized` (ты не залогинен), `403 Forbidden` (залогинен, но нельзя), `404 Not Found` (нет такого), `429 Too Many Requests` (засыпал запросами).
- **5xx — виноват сервер.** `500 Internal Server Error` (упал код на сервере).

> 💡 Разница `401` и `403` — частый вопрос на интервью. `401` = «я не знаю, кто ты»
> (нет/протух токен). `403` = «я знаю, кто ты, но тебе сюда нельзя» (не хватает прав).

## 1. Из чего состоит проект (файлы)

```
step-01-hello-api/
├── HelloApi.csproj         # "паспорт" проекта: версия .NET, NuGet-пакеты
├── Program.cs              # ТОЧКА ВХОДА: настройка + конвейер обработки запроса
├── Controllers/
│   └── PingController.cs   # наши эндпоинты
├── appsettings.json        # конфигурация (настройки, строки подключения и т.п.)
├── Properties/
│   └── launchSettings.json # как запускать локально (порты, окружение)
└── obj/, bin/              # СГЕНЕРИРОВАННОЕ при сборке. В git это не коммитят (см. .gitignore)
```

**`.csproj`** — это XML-файл, описывающий проект. `Sdk="Microsoft.NET.Sdk.Web"`
означает «это веб-проект». `<TargetFramework>net8.0</TargetFramework>` — на какой
версии .NET собираем. Сюда же добавляются NuGet-пакеты (библиотеки) — на Шаге 2
ты впервые добавишь сюда EF Core.

> 📌 `<Nullable>enable</Nullable>` — включает «nullable reference types»: компилятор
> начинает предупреждать, где переменная может быть `null`. `string` — не может быть null,
> `string?` — может. Это защита от самой частой ошибки в мире — `NullReferenceException`.
>
> 📌 `<ImplicitUsings>enable</ImplicitUsings>` — автоматически подключает самые
> частые `using` (System, System.Linq и др.), чтобы не писать их в каждом файле.

## 2. `Program.cs` — две фазы, которые надо понять раз и навсегда

Любой ASP.NET Core проект (и наш, и Favilonia) устроен по одной схеме. Файл читается
сверху вниз и делится строкой `var app = builder.Build();` на две фазы:

### ФАЗА 1 — НАСТРОЙКА: «что приложение умеет»

```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllers();
```

`builder.Services` — это **контейнер зависимостей (DI-контейнер)**. Представь
коробку, в которую мы заранее складываем все «инструменты», которые понадобятся
приложению. `AddControllers()` кладёт в коробку всё, что нужно для работы контроллеров.

> 🔑 **Dependency Injection (внедрение зависимостей)** — концепция №1 на собеседовании.
> Идея: класс **не создаёт сам** то, что ему нужно (`new SomeService()`), а **просит**
> это в конструкторе, и фреймворк подаёт готовое. В Favilonia смотри `GradesController`:
> ```csharp
> public GradesController(AppDbContext db) { _db = db; }
> ```
> Контроллер не создаёт базу сам — он говорит «мне нужен `AppDbContext`», и DI-контейнер
> (та самая «коробка», куда в `Program.cs` положили `AddDbContext`) подаёт его.
> **Зачем так:** так классы не привязаны жёстко друг к другу, и их легко подменять —
> например, в тестах подсунуть фальшивую базу. (Подробно прочувствуешь на Шаге 6.)

### ФАЗА 2 — КОНВЕЙЕР: «как обрабатывается запрос»

```csharp
var app = builder.Build();   // граница фаз
app.MapControllers();
app.Run();
```

Каждый входящий запрос проходит через **middleware pipeline** — конвейер из «прослоек».
Каждая `app.Use...()` — это прослойка, которая может что-то сделать с запросом и
передать дальше (или оборвать). **Порядок строк = порядок прохождения.**

В нашем мини-проекте конвейер почти пустой: только `MapControllers()` — он в конце
смотрит на URL и выбирает нужный метод контроллера.

В Favilonia конвейер уже «взрослый» (`Backend/Favilonia.API/Program.cs`):

```csharp
app.UseHttpsRedirection();                       // 1. перенаправить http -> https
app.UseCors("FrontendPolicy");                   // 2. разрешить запросы с фронта
app.UseMiddleware<ApiExceptionHandlerMiddleware>();// 3. ловить ошибки -> красивый 500
app.UseAuthentication();                          // 4. КТО ты? (читаем JWT-токен)
app.UseAuthorization();                           // 5. МОЖНО ли тебе сюда? (проверяем права)
app.UseRateLimiter();                             // 6. не слишком ли часто стучишься?
app.MapControllers();                             // 7. выбрать контроллер и метод
```

> 🔑 **Порядок здесь критичен.** `UseAuthentication` (выясняем, кто ты) обязан стоять
> ПЕРЕД `UseAuthorization` (решаем, можно ли). Поменяешь местами — авторизация будет
> проверять права у ещё не опознанного пользователя. Это классический вопрос на интервью.

## 3. `PingController.cs` — как запрос находит свой метод

Открой файл — там два эндпоинта с комментариями. Главное, что нужно понять, — **роутинг**
(как URL сопоставляется с методом):

```csharp
[ApiController]
[Route("api/[controller]")]      // базовый адрес: /api/ping  ([controller] = "ping")
public class PingController : ControllerBase
{
    [HttpGet]                    // GET /api/ping
    public IActionResult Get() => Ok(new { message = "pong" });

    [HttpGet("hello/{name}")]    // GET /api/ping/hello/Kenik  ({name} = "Kenik")
    public IActionResult Hello(string name) => Ok(new { message = $"Привет, {name}!" });
}
```

Разбор по частям:

- **`[ApiController]`** — помечает класс как REST-контроллер, включает автоматические
  удобства (валидацию входных данных, привязку параметров из тела/URL/query).
- **`[Route("api/[controller]")]`** — задаёт URL контроллера. `[controller]` — это
  плейсхолдер: ASP.NET берёт имя класса, отрезает слово `Controller` → `Ping` → `ping`.
- **`: ControllerBase`** — базовый класс, дающий помощники: `Ok()`, `NotFound()`,
  `BadRequest()`, `Forbid()`, `CreatedAtAction()`. Каждый возвращает ответ с нужным кодом.
- **`[HttpGet]` / `[HttpGet("hello/{name}")]`** — на какой метод+путь отвечает функция.
  Фигурные `{name}` — это **route-параметр**: значение берётся прямо из URL.
- **`IActionResult`** — тип возврата, означающий «какой-то HTTP-ответ». `Ok(x)` вернёт
  200 + `x` в виде JSON.

> 📌 Это **ровно тот же** механизм, что в `GradesController` Favilonia. Сравни:
> наш `[Route("api/[controller]")]` против тамошнего
> `[Route("api/organizations/{organizationId:guid}/grades")]`. Разница лишь в том, что
> у них в адрес добавлен `{organizationId:guid}` — это и есть начало мультитенантности
> (каждый ресурс живёт «внутри организации»). Дойдём до этого на Шаге 5.

### Путь одного запроса от начала до конца

`GET /api/ping/hello/Kenik` проходит так:

1. Kestrel (веб-сервер внутри .NET) принимает TCP-соединение и парсит HTTP-запрос.
2. Запрос входит в middleware pipeline (у нас он короткий).
3. `MapControllers` (роутер) смотрит: метод `GET`, путь `/api/ping/hello/Kenik`.
   Находит `PingController.Hello`, потому что его шаблон `api/ping/hello/{name}` подходит.
4. Из URL вынимается `name = "Kenik"` и подаётся в аргумент метода (**model binding**).
5. Метод выполняется, возвращает `Ok(...)`.
6. ASP.NET сериализует объект в JSON, ставит статус `200` и `Content-Type: application/json`.
7. Ответ идёт обратно через конвейер к клиенту.

## 4. Запусти и пощупай сам

Из этой папки:

```bash
dotnet run
```

В консоли увидишь строку вида `Now listening on: http://localhost:5xxx`. Возьми этот порт.
В другом терминале (или в браузере для GET):

```bash
curl http://localhost:5xxx/api/ping
curl http://localhost:5xxx/api/ping/hello/Kenik
```

Останови сервер — `Ctrl+C`.

> 💡 Можно запустить на фиксированном порту: `dotnet run --urls "http://localhost:5099"`.
