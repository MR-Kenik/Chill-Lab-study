# Словарь терминов

Короткие объяснения «на пальцах». Термины, которые реально спрашивают на собеседовании
бэкенд-разработчика, помечены ⭐.

---

### ASP.NET Core
Фреймворк от Microsoft для написания веб-приложений и API на C#. На нём построена
вся серверная часть Favilonia.

### ⭐ DI / Dependency Injection (внедрение зависимостей)
Подход, при котором класс не создаёт нужные ему объекты сам (`new Foo()`), а получает
их «извне» — обычно через конструктор. Фреймворк держит «контейнер» зарегистрированных
сервисов и подаёт их автоматически. Зачем: слабая связанность, лёгкая подмена в тестах.
*Пример:* `public GradesController(AppDbContext db)` — контроллер просит базу, а не создаёт.

### ⭐ Middleware / pipeline (конвейер)
Цепочка «прослоек», через которую проходит каждый HTTP-запрос по пути к контроллеру и
обратно. Каждая может обработать запрос и передать дальше или оборвать. Порядок важен.
*Примеры прослоек:* аутентификация, CORS, обработка ошибок, rate limiting.

### ⭐ Controller (контроллер)
Класс с методами-обработчиками (actions), отвечающими на HTTP-запросы. Каждый метод
привязан к методу+пути (`[HttpGet]`, `[HttpPost("...")]`).

### Action / эндпоинт
Один метод контроллера = один обработчик одного запроса. «Эндпоинт» — это адрес+метод,
по которому он доступен (напр. `GET /api/ping`).

### Routing (роутинг)
Механизм сопоставления URL запроса с нужным методом контроллера. Настраивается
атрибутами `[Route]`, `[HttpGet]` и плейсхолдерами вроде `[controller]`, `{id}`.

### Route-параметр vs query-параметр
- **Route:** часть пути — `/api/ping/hello/{name}` → `.../hello/Kenik`.
- **Query:** хвост после `?` — `/api/ping/sum?a=2&b=3`.

### ⭐ DTO (Data Transfer Object)
Отдельный класс для данных, которые «ходят» между клиентом и API (на вход — Request,
на выход — Response). Нужен, чтобы НЕ отдавать наружу сущности базы целиком (безопасность
+ производительность). *Пример:* `GradeResponse` вместо самой `Grade`.

### ⭐ Entity (сущность)
C#-класс, который EF Core отображает в таблицу базы данных. *Пример:* `Grade`, `User`,
`Organization`. Свойства класса = колонки таблицы.

### ⭐ ORM / EF Core (Entity Framework Core)
ORM = Object-Relational Mapper: библиотека, превращающая объекты C# в строки таблиц и
LINQ-запросы в SQL. EF Core — это конкретный ORM от Microsoft. *Пример:*
`_db.Grades.Where(x => x.Value > 3)` → EF сгенерирует `SELECT ... WHERE value > 3`.

### ⭐ DbContext
Главный класс EF Core, представляющий «сессию работы с базой». Содержит `DbSet<T>` —
по одному на каждую таблицу. *Пример:* `AppDbContext` с `DbSet<Grade> Grades`.

### ⭐ Migration (миграция)
Сгенерированный код, описывающий изменение схемы базы (создать таблицу, добавить колонку).
Применяется к базе командой. Позволяет эволюционировать структуру БД версионно, как код.
*В Favilonia:* папка `Migrations`, по файлу на каждое изменение.

### CRUD
Create, Read, Update, Delete — четыре базовые операции над данными. Обычно ложатся на
POST, GET, PUT, DELETE соответственно.

### LINQ
Язык запросов внутри C# (`.Where()`, `.Select()`, `.OrderBy()`). EF Core переводит
LINQ-запросы в SQL.

### ⭐ Authentication (аутентификация) vs Authorization (авторизация)
- **Authentication** — «КТО ты?» (проверка логина/токена). Ошибка → `401`.
- **Authorization** — «можно ли ТЕБЕ это?» (проверка прав/ролей). Ошибка → `403`.
Сначала всегда аутентификация, потом авторизация.

### ⭐ JWT (JSON Web Token)
Подписанный токен, который сервер выдаёт при логине, а клиент потом шлёт в каждом запросе
(заголовок `Authorization: Bearer ...`). Внутри — «claims» (id пользователя, роль, организация).
Сервер проверяет подпись и доверяет содержимому, не лазая в базу на каждый запрос.

### ⭐ Claim
Пара «ключ-значение» внутри токена — факт о пользователе. *Примеры:* `userId`, `role`,
`organizationId`. В Favilonia читаются через `User.GetUserId()`, `User.GetOrganizationId()`.

### Access token / Refresh token
- **Access token** — короткоживущий (≈60 мин) JWT для доступа к API.
- **Refresh token** — долгоживущий (≈7 дней), хранится в базе, нужен чтобы получить новый
  access-токен без повторного ввода пароля. При использовании «прокручивается» (ротация).

### Hash пароля
Пароли никогда не хранят в открытом виде. Хранят их **хеш** — необратимое преобразование.
При логине хешируют введённый пароль и сравнивают хеши. *В Favilonia:* `PasswordHasher`.

### ⭐ Мультитенантность (multi-tenancy)
Архитектура, где один экземпляр приложения обслуживает много клиентов (tenants —
здесь это организации), при строгой изоляции их данных. Ключевая особенность Favilonia.

### Tenant (тенант)
Один изолированный клиент системы. В Favilonia тенант = организация (`Organization`),
у каждой записи есть `OrganizationId`.

### ⭐ Soft delete (мягкое удаление)
Запись не удаляется из базы физически, а помечается флагом `IsDeleted = true` и
скрывается из обычных запросов (через global query filter). Можно восстановить/проверить аудит.

### Policy (политика авторизации)
Именованное правило доступа. *В Favilonia:* `SameOrganization` (токен совпадает с
организацией в URL), `AdminOnly` (нужна роль Admin), `SuperAdmin`.

### Status code (код ответа HTTP)
Число, говорящее об исходе: 2xx — успех, 4xx — ошибка клиента, 5xx — ошибка сервера.
Минимум назубок: 200, 201, 204, 400, 401, 403, 404, 429, 500.

### Kestrel
Встроенный в .NET веб-сервер, который реально слушает порт и принимает HTTP-запросы.
Запускается строкой `app.Run()`.

### ⭐ Интеграционный тест
Тест, проверяющий несколько слоёв вместе (контроллер + БД + авторизацию) на «живом»
приложении, поднятом в памяти. Противоположность — unit-тест (проверяет один класс изолированно).
